PART C · 이름이 사는 곳 — CH 11

클로저

함수 안에서 함수를 만들어 밖으로 돌려주면, 그 안쪽 함수는 자기가 태어난 곳의 이름(바깥 함수의 지역 이름)을 기억해요. 바깥 함수가 끝나 사라진 뒤에도요. 이걸 클로저라고 해요. 핵심 미묘함 — 클로저는 그 이름의 값을 찍어 저장하는 게 아니라 이름(변수) 자체를 기억해, 호출되는 순간의 값을 봐요(늦은 바인딩). 이걸 모르면 그 악명 높은 "루프 클로저 함정"에 빠져요.

Step 00 — Remembering Its Birthplace

안쪽 함수가 바깥 이름을 기억한다

make_adder(n) 가 lambda x: x + n 을 돌려준다고 해요. 이 안쪽 함수는 바깥 n 을 쓰죠. add5 = make_adder(5) 로 받으면, make_adder 는 이미 끝났는데도 add5 는 n=5 를 기억하고 있어요. 안쪽 함수가 바깥 방을 붙잡고 있어서, 그 방은 평소처럼 사라지지 않고 함께 살아남아요. 그게 클로저예요.

→ 돌려받은 안쪽 함수가 바깥 방(이름들)을 붙잡아 살려둠 = 클로저.

Step 01 — Names, Not Values

값을 찍는 게 아니라 이름을 기억한다

결정적 미묘함 — 클로저는 만들어질 때 n 의 값을 사진 찍어 박제하는 게 아니에요. 이름 n 그 자체를 기억해, 안쪽 함수가 실제로 호출될 때 그 이름의 값을 읽어요(늦은 바인딩). 그래서 만든 뒤에 바깥에서 n 을 바꾸면, 클로저가 보는 값도 바뀌어요. 버튼으로 "이름을 기억 → 호출 시 값 조회"를 따라가 보세요.

→ 클로저는 변수(이름)를 캡처, 호출 시점에 값을 읽음(늦은 바인딩). 값 박제가 아님.

Step 02 — The Loop Closure Trap

루프 안에서 만든 함수들이 다 똑같다?

고전 함정 — 루프에서 함수를 여러 개 만들어요: [lambda: i for i in range(3)]. 기대는 0·1·2를 돌려주는 세 함수죠. 그런데 셋 다 2를 돌려줘요! 셋이 같은 변수 i 를 기억하는데, 호출은 루프가 끝난 뒤라 그때 i 는 이미 마지막 값 2거든요(늦은 바인딩). 고치려면? CH07의 그 트릭 — 기본 인자(lambda i=i: i)로 만들 때 값을 박아넣어요. 토글로 비교.

→ 같은 변수 i 공유 → 다 마지막 값. 회피: 기본 인자로 def 때 값 고정(CH07).

Step 03 — Takeaways

정리

클로저는 안쪽 함수가 바깥 함수의 이름(변수)을 붙잡아 기억하는 거예요 — 값을 박제하는 게 아니라 호출 시점에 값을 읽죠(늦은 바인딩). 루프에서 만든 클로저들이 같은 변수를 공유해 다 같은 값을 내는 함정도 여기서 나오고, 회피책은 기본 인자(CH07)예요. 그런데 — 잠깐, 함수를 돌려주고, 변수에 담고, 리스트에 넣었죠? 함수가 그냥 값처럼 다뤄졌어요. 그게 가능한 이유 — 파이썬에선 함수도 객체예요. 다음 PART.
